# PocketGraphRAG：我是如何用 3000 行代码打造一个轻量级 GraphRAG 框架的

> 从 0 到 1，一个人，两周时间，一个可以跑在你笔记本上的 GraphRAG 框架。

## 前言

做 GraphRAG 有一段时间了，用过微软的 GraphRAG，也试过 LightRAG，但总觉得：

- **太重了** — 要装 Neo4j，要配置一堆东西，小项目根本用不上
- **太复杂了** — 一堆概念，上手成本高
- **不接地气** — 英文文档为主，中文支持差
- **可视化弱** — 跑出来的结果看不见摸不着

于是我决定自己写一个：**PocketGraphRAG** — 装在口袋里的 GraphRAG。

目标很明确：
- ✅ **pip install 就能用**，不依赖任何外部数据库
- ✅ **中文友好**，BGE 中文 Embedding + 中文 Prompt
- ✅ **看得见**，交互式图谱可视化，检索路径透明
- ✅ **跑得动**，笔记本 CPU 就能跑
- ✅ **可扩展**，留好接口，想加什么加什么

两周后，它长这样：131 个单元测试、11 种图算法、5 种检索模式、REST API、Docker 部署、Web UI、多模态基础框架...

今天聊聊它的设计思路和实现细节。

## 一、核心设计：为什么是「实体级切块」？

传统 RAG 的思路是：把文档切成一块块（chunk），每块转成向量存起来。查询时，把问题也转成向量，找最相似的几块。

这个思路的问题是：**切块是按字数切的，和语义无关**。

比如「盗梦空间的导演是... 剧情简介是... 诺兰的其他作品是...」，如果一刀切在「剧情简介」中间，上下文就碎了。

PocketGraphRAG 的思路不一样：**先从文档里抽出知识图谱（实体 + 关系），再按「实体」来组织文本块**。

```
传统方式：
  [文本块1] [文本块2] [文本块3]  ← 按字数切，语义不完整

我们的方式：
  实体A → 文本块（所有和A相关的内容）
  实体B → 文本块（所有和B相关的内容）
  实体C → 文本块（所有和C相关的内容）
```

好处是：
1. **上下文完整** — 一个实体的所有信息聚在一起
2. **可解释性强** — 你知道答案是从哪个实体来的
3. **可以推理** — 有了实体和关系，就能做图上的推理

## 二、检索架构：LightRAG 风格的双层检索

光有实体级切块还不够，怎么从图里找到最相关的知识？

我参考了 LightRAG 的思路，做了**双层检索**：

### 2.1 Local Search（局部检索）：从实体出发

```
用户问题 → 匹配到种子实体 → BFS 扩展邻域 → 找到相关实体
```

举个例子：你问「盗梦空间讲了什么？」
- 先匹配到「盗梦空间」这个实体
- 然后沿着「导演」「主演」「类型」这些关系扩展
- 找到「诺兰」「莱昂纳多」「科幻悬疑片」这些相关实体
- 把这些实体对应的文本块都拿出来

### 2.2 Global Search（全局检索）：从关系出发

```
用户问题 → 匹配到相关关系类型 → 找到所有满足该关系的实体对
```

比如你问「哪些农药可以治病？」
- 先匹配到「化学防治」「生物防治」这些关系类型
- 然后找出所有「农药 → 防治 → 病害」这样的三元组
- 把相关实体和文本块都拿出来

### 2.3 Mix 模式：1+1>2

两个模式各有侧重：
- Local 适合**具体实体**的问题（「盗梦空间的导演」）
- Global 适合**类型性**的问题（「有哪些生物防治方法」）

Mix 模式把两个结果融合在一起，用 **RRF（Reciprocal Rank Fusion）** 算法做融合 — 这是业界公认的高效融合算法，不需要归一化分数，直接用排名计算。

```python
# RRF 公式
score(d) = Σ 1 / (k + rank_i(d))
# k 通常取 60
```

### 2.4 Pagerank 加权：重要的实体排前面

光有相似度还不够，**实体本身的重要性**也应该影响排序。

比如「盗梦空间」这个实体，在图里连接了很多其他实体（导演、主演、类型...），它就是个「枢纽节点」。

我把 Pagerank 算法加进去了：

```
最终分数 = 向量相似度 + Pagerank 重要性 × 权重
```

实测下来，对回答质量提升很明显 — 重要实体更容易被检索到。

## 三、实现细节：那些踩过的坑

### 3.1 实体名称不一致怎么办？

同一个东西可能有不同叫法：
- 「苏云金杆菌（Bt）制剂」 vs 「苏云金杆菌制剂」
- 「用药量」 vs 「农药使用量」

如果用子串匹配，根本对不上。

**解决方案：Entity Embedding Matching**

把所有实体名转成向量（用 BGE 模型），查询时也把问题转成向量，算相似度。

```python
# 伪代码
entity_embeddings = model.encode(all_entity_names)
query_embedding = model.encode(query)
similarities = cosine_similarity(query_embedding, entity_embeddings)
matched_entities = [entities[i] for i in top_k_indices]
```

这样即使叫法不一样，语义相近也能匹配上。

### 3.2 反向关系怎么处理？

知识图谱里的关系是有方向的：
- 「诺兰 → 导演 → 盗梦空间」

但用户可能会反过来问：「盗梦空间是谁导演的？」

如果只有正向关系，这个问题就答不好。

**解决方案：自动推断反向链接**

- 关系名里有「防治/治疗/包含/属于」这类动词的，自动建立反向链接
- 用户也可以通过环境变量配置哪些关系需要反向

### 3.3 Gradio 里的 ECharts 不执行？

图谱可视化用的 ECharts，放在 Gradio 的 HTML 组件里，结果脚本根本不执行。

查了半天发现：Gradio 的 HTML 组件为了安全，会把 `<script>` 标签里的内容禁掉。

**解决方案：base64 data URL + iframe**

```python
# 把 HTML 内容 base64 编码
html_b64 = base64.b64encode(html_content.encode()).decode()
iframe_src = f"data:text/html;charset=utf-8;base64,{html_b64}"

# 用 iframe 嵌进去
return f'<iframe src="{iframe_src}"></iframe>'
```

完美解决。

## 四、性能优化：从 1 秒到 1 毫秒

最初 Pagerank 是用 Python 循环写的，500 个节点就要 1 秒多。

优化后（NumPy 向量化），500 个节点只要 **1 毫秒**，提升了 1000 倍。

关键思路：把「节点-节点」的遍历，变成「边列表的矩阵运算」。

```python
# 优化前：Python 循环
for i in range(n):
    for j in adj[i]:
        new_pr[j] += damping * pr[i] / len(adj[i])

# 优化后：NumPy 向量化
edge_contrib = pr[src] / out_degree[src]
np.add.at(new_pr, dst, damping * edge_contrib)
```

同理，个性化 Pagerank、社区发现、最短路径，都用类似的思路优化。

现在的性能（500 实体，2000 边）：

| 算法 | 耗时 |
|------|------|
| Pagerank | 0.001s |
| 个性化 Pagerank | 0.0008s |
| 度中心性 | <0.0001s |
| 社区发现 | 0.016s |
| 介数中心性 (k=50) | 0.035s |

完全够用。

## 五、架构总览

```
用户输入
   ↓
查询理解 (Multi-hop 分解)
   ↓
┌─────────────┬─────────────┐
│  向量检索    │  KG 双层检索  │
│  (FAISS+BGE) │ (Local+Global)│
└─────────────┴─────────────┘
   ↓
Pagerank 加权 + RRF 融合
   ↓
重排序 & Top-K
   ↓
LLM 生成回答（流式输出）
```

整个流水线，每一步都是可配置、可替换的。
